### TEMPLATING ENGINE


Templating engines are used to render a static file dynamically based on the context of the request and user. It lets web apps render dynamic HTML by mixing static content with runtime data.

For example: A header that shows the username when the user is logged in. When no user is logged in, the header might say "Hello, Guest"; however, as soon as a user logs in, the header will change to "Hello, Username". 
Its like a mail merge, we write a template once, and the engine fills in variables per request.
It allows developers to centralize the location of reusable content and also to further separate the view from the Model-View-Controller.
Jinja2 is Python's most popular templating engine, used heavily in frameworks like Flask and Frappe/ERPNext 

A templating engine uses delimiters so developers can tell the engine where a template block starts and ends. The most common delimiters are **expressions and statements**. 

- `{{ expression }}` : evaluates and outputs a value
- `{% statement %}`  : performs an action (loop, condition, etc.)
 

## SERVER SIDE TEMPLATE INJECTION

Server-side template injection (SSTI) occurs when an attacker is able to inject code into a server-side template, which is then executed by the server. This can occur when an application uses user input to construct a template without properly validating or sanitizing the input.

The vulnerable pattern looks like this:
 
```python
template = Template("Hello " + user_input)
render(template)
```

This code is vulnerable to Server-Side Template Injection (SSTI) because user input was directly concatenated to the template. There is never a good reason to concatenate user input to a template. Instead, the user input should become a variable that is passed to the template like:

```python
# user input is passed as a variable value : Safe
template = Template("Hello {{ name }}")
template.render(name=user_input)
```
If we submit `{{ 7*7 }}`, the server renders `Hello 49`, it evaluated the expression instead of treating the input as plain text. That's SSTI.

A templating engine commonly uses its own syntax separate from the languages it was built in, but with many ties back into it. As an example, to get the length of a string in Python, we use len function:
```python
len("hello!")
```
Jinja uses the "|" character to pipe a variable into the length filter. However, this filter will still run the Python len function. While Jinja might use a separate syntax for writing expressions and statements, the underlying "kernel" is still Python.

If an application gives us the ability to inject into templates, we might be able to escape the "sandbox" of the templating engine and run system-level code. Some templating engines contain direct classes to execute system-level calls

## Exploit

Jinja2 is built on top of Python. When Jinja renders a template, it runs in a restricted environment or a sandbox,designed to allow display logic while blocking dangerous operations like file reads or shell execution.

The problem is that Jinja's sandbox is not a true isolation boundary. Every value we interact with inside a Jinja template is a real Python object. And every Python object carries built in attributes that let us ask it what it is, where it came from, and what else exists in the same Python process.

There are objects that will be always accessible from the sandboxed env, these are:
`[],'',(),dict,config,request`
from these objects we need to get to the class: `<class 'object'> `in order to try to recover defined classes. This is because from this object we can call the __subclasses__ method and access all the classes from the non-sandboxed python env.

In order to access that object class, you need to access a class object and then access either  __base__, __mro__()[-1] or .mro()[-1]. And then, after reaching this object class we call __subclasses__().

One of the common payload that is used to exploit Jinja:
`{{ ''.__class__.__mro__[2].__subclasses__()[40]('/etc/passwd').read() }}` 

- An empty string is created with the two single-quote characters. Next, the __class__ attribute returns the class to which the string belongs, the str class.

- Next the payload uses the __mro__ attribute ("Method Resolution Order"). When Python looks up an attribute, it searches through classes in a specific order. MRO (Method Resolution Order) is the ordered tuple that defines that search path

# Inheritance

In Python, a class can be built on top of another class. The child inherits all attributes and methods from the parent. Consider this example:
 
```python
class Food:
    calories = 100
 
class Fruit(Food):        # Fruit inherits from Food
    fructose = 2.0
 
class Strawberry(Fruit):  # Strawberry inherits from Fruit and Food
    ripeness = "Ripe"
 
s = Strawberry()
s.calories    #  100    inherited from Food
s.fructose    #  2.0    inherited from Fruit
s.ripeness    #  "Ripe" its own
```
Strawberry never defined `calories`,it got it by inheriting through the chain `Strawberry -> Fruit -> Food`.

```python
Strawberry.__mro__
#  (<class 'Strawberry'>, <class 'Fruit'>, <class 'Food'>, <class 'object'>)
```
 
So the goal of using `__mro__` is to reach `object` — the root of everything. In Python 3, `str` inherits directly from `object`, so it sits at index `[1]`. In Python 2.7, there is an extra class `basestring` in between, pushing `object` to index `[2]`.


- Next we call `__subclasses__()` on the `object` class. Since `object` is the parent of everything in Python, this returns **every class currently loaded in the Python process** — built-ins, standard library modules, and everything the web framework imported at startup.
 
This is the treasure chest. Somewhere in this list are classes that can read files, run shell commands, or make network calls.
 
> **Important:** The list is dynamic. It only contains classes alive in memory at that moment. The indices vary between targets depending on what the application has loaded at runtime.

In Python 2.7, the returned item in the 40th index is the file type. The payload is using the file type, passing in the file to be read (/etc/passwd), and running the read method. In Python 2.7, we can read the /etc/passwd file.

## The Full Chain for Python 2.7
 
| Step | Code | What It Does |
|---|---|---|
| 1 | `''` | Start with any Python object |
| 2 | `.__class__` | Get the `str` class |
| 3 | `.__mro__` | Get the inheritance chain: `(str, basestring, object)` |
| 4 | `[1]` or `[2]` | Grab `object` (index depends on Python version) |
| 5 | `.__subclasses__()` | List every class loaded in the process |
| 6 | `[40]` | Grab the `file` class (Python 2.7 only) |
| 7 | `('/etc/passwd')` | Open the target file |
| 8 | `.read()` | Read and return its contents |

## Bypassing filters

The application has a filter blocking .__ (dot followed by double underscore). So any direct attribute access like .__class__ gets caught and stripped or rejected.
Jinja2 has its own syntax for getting object attributes that doesn't require the dot Python uses. The attr() filter lets you access an attribute by passing its name as a string argument, using '|': `object|attr("attribute_name")`.
So instead of `'ssti'.__class__`, we can  write `'ssti'|attr("__class__")`.

To avoid `.__` appearing anywhere in our payload, first store all attribute names as variables and then chain them using `attr()`:
 
```jinja
{% set string = "ssti" %}
{% set class = "__class__" %}
{% set mro = "__mro__" %}
{% set subclasses = "__subclasses__" %}
```
 
At this point, nothing suspicious has happened from the filter's perspective. We've assigned some plain strings to variables and the literal `.__` never appears anywhere in the template text.
 
Then chain them using `attr()`:
 
```jinja
{{ string|attr(class)|attr(mro) }}
```
 
Which is equivalent to `"ssti".__class__.__mro__` and so on till we reach  `__subclasses__()` and a target class.
